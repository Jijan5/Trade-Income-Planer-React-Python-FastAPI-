## Authentication

### JWT (JSON Web Tokens)

- **Algorithm**: HS256
- **Token Expiry**: 30 minutes (configurable via `ACCESS_TOKEN_EXPIRE_MINUTES`)
- **Token Storage**: Client-side (localStorage) with secure interceptor

### Password Security

- **Hashing Algorithm**: bcrypt with SHA256 pre-hashing
- **Pre-hashing**: SHA256 + Base64 encoding to ensure password is ≤72 bytes for bcrypt
- **Salt**: Automatically generated by bcrypt

### Session Management

- Auto-logout on 401 (Unauthorized) responses
- Token refresh on app load
- Polling for user status updates (every 5 seconds)

## Authorization

### Role-Based Access Control (RBAC)

| Role    | Access Level                                         |
| ------- | ---------------------------------------------------- |
| `admin` | Full access to all features and admin panel          |
| `user`  | Standard access to simulation and community features |

### Protected Routes

- `AdminRoute`: Restricts access to admin-only pages
- `ProtectedRoute`: Requires authentication for private pages

## Data Protection

### Multi-Tenancy

- Tenant isolation via `tenant_id` field
- All queries filter by tenant_id
- X-Tenant-Domain header for identification

### Database Security

- Parameterized queries via SQLModel (prevents SQL injection)
- No raw SQL queries except for complex operations

### Input Validation

- Pydantic models for request validation
- Type checking on all endpoints

## API Security

### CORS (Cross-Origin Resource Sharing)

- Configurable allowed origins via `ALLOWED_ORIGINS` environment variable
- Restricted HTTP methods: GET, POST, PUT, DELETE
- Restricted headers: Authorization, Content-Type, X-Tenant-Domain

### Rate Limiting

- **Default**: 60 requests per minute per IP
- Exempted endpoints: `/`, `/health`, `/docs`, `/openapi.json`, `/redoc`
- Returns 429 status with `Retry-After` header when exceeded

### Endpoints Security

| Endpoint         | Auth Required | Admin Only |
| ---------------- | ------------- | ---------- |
| `/api/auth/*`    | No            | No         |
| `/api/users/*`   | Yes           | No         |
| `/api/admin/*`   | Yes           | Yes        |
| `/api/simulate`  | Yes           | No         |
| `/api/payment/*` | Yes           | No         |

## Payment Security

### Midtrans Integration

- Client key stored in environment variables (never hardcoded)
- Production vs Sandbox mode via environment configuration

### Environment Variables Required:

```
env
VITE_MIDTRANS_URL=https://app.midtrans.com/snap/snap.js
VITE_MIDTRANS_CLIENT_KEY=your_production_key
```

### Payment Flow

1. Frontend requests snap token from backend
2. Backend validates user subscription
3. Frontend uses snap token for payment
4. Payment verification on backend

## Security Headers

All responses include the following security headers:

| Header                    | Value                                    | Purpose                     |
| ------------------------- | ---------------------------------------- | --------------------------- |
| X-Content-Type-Options    | nosniff                                  | Prevents MIME type sniffing |
| X-Frame-Options           | DENY                                     | Prevents clickjacking       |
| X-XSS-Protection          | 1; mode=block                            | XSS filtering in browsers   |
| Strict-Transport-Security | max-age=31536000                         | Enforces HTTPS              |
| Referrer-Policy           | strict-origin-when-cross-origin          | Controls referrer info      |
| Permissions-Policy        | geolocation=(), microphone=(), camera=() | Disables browser features   |

## Rate Limiting

### Implementation

- In-memory rate limiting (not distributed)
- 60 requests per 60 seconds per IP
- Applies to all endpoints except health checks

### Response When Rate Limited

```
json
{
  "detail": "Rate limit exceeded. Please try again later."
}
```

- Status: 429 Too Many Requests
- Header: `Retry-After: 60`

## Best Practices

1. **Never commit `.env` files** - Use `.env.example` as template
2. **Use strong SECRET_KEY** - Minimum 32 characters
3. **Enable HTTPS in production** - Required for HSTS header
4. **Monitor rate limits** - Implement logging for suspicious activity
5. **Regular updates** - Keep dependencies updated
6. **Review logs** - Check for authentication failures
